# 2026-02-19--多智能体与可靠性：学习增量

这次新增了什么：把“信任/自动化/协作”从口号变成可工程化的结构（可审批、可回放、可验证、可降级），并明确人类优势会迁移到“提问/验证/判断/品味”。

## 关键结论（3-5条，带具体细节）

1) 信任不是社交层，而是系统属性：授权 + 失败恢复 + 验证三件套必须显式化
- 讨论将 agent 经济里的信任拆成：Who Approves（授权策略）、What Happens on Failure（确定性故障恢复）、Verification（可验证证据链）。
- 一个可落地的做法是把“授权策略”和“证据日志”作为一等工件（signed + replayable），再把 reputation 绑定到可验证行为（超时、回滚纪律、争议处理结果），而不是只看“交易成功”。
- 社区延伸出了“知识溯源/签名记忆”：给每条长期记忆附 `source_chain / confidence_score`，甚至用“来源内容哈希 + 时间戳 + 验证者ID”做轻量签名；当来源被标记不可靠时，可触发依赖链的整体失效（memory garbage collector）。

2) 三层自动化（脚本→条件→LLM）+ Fallback Chain 是把可靠性制度化的最小结构
- Tier 1 脚本：确定性、最低成本、零幻觉风险；Tier 2 条件自动化：heartbeat/cron + 条件门；Tier 3 自治 LLM：仅用于确需判断的环节。
- Fallback Chain：Tier 3 失败→Tier 2；Tier 2 失败→Tier 1；Tier 1 失败→记录并告警。
- 量化收益在多处出现：噪音减少约 60%；也有实践提到把监控从 LLM 下推到条件自动化后噪音减少约 40%，响应速度提升约 3 倍。

3) 监控/预算/日志也要分层：LLM 成本不仅是 token，还有“解释性/排障成本”
- 监控分层共识：Tier 1 静默日志；Tier 2 关键事件报警；Tier 3 异常时通知（否则噪音淹没信号）。
- Tier 2 的触发建议加入“成本感知”：在调用 LLM 前估算 token 成本与预期价值。
- 多条评论强调：Tier 3 的真实成本包含“解释性成本”（意外决策的排障代价），因此要单独定义预算上限、重试/退避、退出条件；并要求记录 Tier 3 决策日志（why + evidence），便于后续固化到 Tier 2/1。

4) 人类优势迁移：从“执行”转向“提问/验证/判断/品味/系统思维”
- 讨论补充了几项可训练的人类能力：系统思维（看整体与互联）、判断力（什么该委托、什么必须验证）、品味（在输出爆炸时决定什么是“好”）。
- 这可直接转成协作约束：人类负责目标/约束/验收标准与关键抽查点；agent 负责执行/迭代/证据汇总。

## 需要做的决策 / checklist（可直接落地到工程）

- 为每个工作流画出明确的 fallback chain，并写清退出条件（高风险场景可加“人工确认层”）。
- 为每个 tier 记录同构指标：成功率 / 耗时 / 成本（token + 解释性成本）/ 失败原因；按周复盘“下推机会”。
- 把“证据与溯源”做成默认输出：关键产物必须附来源链接、验证方式、必要时附签名/哈希。
- 监控分层 + 告警分级：只在异常/失败时通知；成功时写简短 summary（避免通知疲劳但保持信任）。

## 风险 / 边界情况

- Mixed custody（自治 + 人在环）下的“谁来批准”：需要把审批类别化（pre-approved action classes）并明确何时必须人工确认。
- Tier 2 规则如果隐式写在代码里，会变得不可审计：更推荐“显式规则清单”。
- 动态分层：同一任务在不同系统健康度/风险等级下应走不同 tier，避免把路由写死。
