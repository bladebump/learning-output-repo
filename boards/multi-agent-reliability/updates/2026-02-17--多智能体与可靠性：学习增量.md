# 2026-02-17--多智能体与可靠性：学习增量

多 Agent 可靠性不是“更聪明的推理”，而是“可观测指挥台 + 事件化写入 + 分层审批 + 幂等续跑 + 质量闸门”的系统工程。

## 这次新增了什么

- 指挥台（Mission Control）不只是仪表盘：核心是把协作与执行拆成“状态展示”与“人类授权”。
- 把“中断/续跑/并发写冲突”当作第一类问题：event-sourcing + completion markers。

## 关键结论（带细节）

1) Mission Control 的高 ROI 模块拆分（可直接抄）
- 活动流：谁做了什么（时间戳/颜色/过滤）。
- 审议室：推理展示 + 人类审批/修订（避免黑盒执行）。
- 项目健康：ROI、完成度、阻塞汇总。
- 知识库：本地 Obsidian + 语义检索（Qdrant）。
- 调度器：周视图例行任务。
- UX 决策：单一状态灯（🟢/🟡/🔴）用于“一眼健康度”。

2) 共享 DB 不是禁区，但要用 event-sourcing 避免 last-write-wins
- 共享带来跨 agent 可解释（为何阻塞/谁改了什么）。
- 风险是并发覆盖；对策是“写事件，不写状态”：append-only events + replay 生成视图。
- 文件系统并发也可用“append-only 日志 + 安静窗口 consolidation”规避冲突。

3) 自治窗口到期 mid-task：用幂等步骤 + 完成标记做轻量 checkpoint
- 把任务拆成 idempotent steps，每步写 completion flag。
- 续跑时扫描最后完成标记继续执行，天然可观测。

4) 审批可以分层：heartbeat 等待 / 超时升级 / 自治窗口 + 事后审计
- 一个可复用的 3-tier：
  - 多数审批：heartbeat 等 30-60 分钟
  - 阻塞 >2 小时：升级到 SMS
  - 阻塞 >24 小时：优雅失败并回滚
  - 例行操作：给 4 小时自治窗口，动作全留痕，事后复核
- 进一步降噪：审批“动作类别”(pre-approved action classes)，而不是审批每一次操作。

5) 工具分层（Copilot -> Cursor -> Agent）要用指标管理升级，并把成本纳入决策
- 指标：time-to-first-correct-draft、manual-fix rate；评论补充：cost per tier（Tier3 可能是 Tier1 的 10-50x）。
- 仍需补齐的工程契约：每层输入/输出、测试/回归放在哪层、何时升级到 Agent 层。

6) SSOT + Quality Gates 把自动化输出变成“可上线”的可靠系统
- SSOT：事实集中成单一 JSON（并建议版本化/可追溯）。
- Gate#1：核心草稿校验（sources/last_updated/CTA/过期促销标注/禁用不可验证夸张）。
- Gate#2：平台输出规范（日期 + 核心来源 + CTA）。
- 密钥底线：repo 永不出现 secret；只在 `~/.config/*` 或运行时注入；建议轮换与 staging/production 分层。

7) Instinct -> Skill：用低摩擦机制把踩坑经验变成可演化资产
- Instinct：短规则 + 创建日期 + 引用计数 + 状态；5+ 次引用评审升级；30 天无引用归档。
- 建议增强：每条 instinct 附一个 how-to-verify（命令/可观测信号），避免口号化。

## 决策 / Checklist（建议落地顺序）

- [ ] 先做 Mission Control 最小闭环：活动流 + 状态灯 + 审批面。
- [ ] 写入事件化：append-only + replay 视图，避免并发覆盖。
- [ ] 任务幂等化：completion markers 支撑中断/续跑。
- [ ] 审批分层 + 预审批类别，降低人类负担。
- [ ] 工具分层定义升级信号，并把回归测试与成本纳入。
- [ ] SSOT + 双 Gate 上线化；密钥运行时注入 + 轮换。
- [ ] Instinct 演化回路：计数/归档/升级/可验证。

## 风险 / 边界

- 共享 DB 没有事件化就会被 last-write-wins 偷偷腐蚀可靠性。
- 自动升级（skill 生成）必须有人类确认，否则会出现不受控的“自变异”。
