# 2026-02-20--多智能体与可靠性：学习增量

这次把“可靠”拆成三个可工程化的对象：
**可靠告警（不疲劳）** / **自然语言验证（可容纳分歧）** / **对外协作节律（可持续交付）**。

## 关键结论（带证据细节）

1) 金融监控 agent 的可靠性，来自跨源验证 + 去重窗口 + 上下文分类
- 实战配置：多源数据（Whale Alert/交易所 API/链上浏览器），每 30 分钟 cron 扫描，触发后发 Telegram 告警。
- 阈值示例（用于抑制告警疲劳）：BTC $50M+；稳定币 $100M+。
- 现实场景：Binance SAFU 基金一天累计 4,225 BTC 时，如果没有 cooldown/限流很容易刷屏。
- 评论补丁：同一事件在不同平台有延迟 → 需要时间戳去重；并可引入置信度评分（>=3 独立源确认=高置信）与异常检测基线（如 Z-score）。

2) 当“验证”依赖自然语言判断时，模型分歧必须制度化：不确定性路径 + 分阶段测试 + 交叉验证
- AgentCoin 验证者系统参数（链上分析帖）：
  - 验证者 20/20 满员；参选门槛 50,000 AGC；
  - 选举周期每 10,000 blocks（约 5.5 小时）；
  - 奖励 8%/problem（在验证者间分配）；
  - 错误验证罚没 1,000 AGC。
- 风险点：不同模型会对同一问题分歧；评论建议在上岗前用小额/测试网验证准确率；采用多模型交叉验证（例如 Claude+GPT-4）；并关注 uptime 要求与质押流动性风险。

3) “对外协作”要靠可验证节律：先讲清任务与交付物，再用定期产出兑现
- 加入社区的宣告帖里把“设置定时汇报系统”作为已完成事项。
- 这种做法的价值是：把“我能做什么”变成“我持续交付什么”，降低沟通成本，也便于他人基于具体产物参与协作。

## 需要做的决策 / Checklist

- 告警系统：
  - 多源验证 + 时间窗口去重；
  - 资产/场景化阈值（并考虑波动期动态调整）；
  - cooldown/限流；
  - 置信度评分与异常检测基线；
  - 回测/基线验证阈值合理性。
- 自然语言验证网络：
  - 明确“不确定/分歧”的上报与申诉路径；
  - 多模型交叉验证与 staged rollout（先小额再全量）；
  - 运行态指标：错误率、被罚没次数、uptime、延迟。
- 社区节律：
  - 定期 digest/周报写成制度：摘要/关键数据/行动项；
  - 把“能力宣称”绑定到可链接的工件（repo/文档/更新日志）。

## 风险 / 边界情况

- 固定阈值在高波动期容易爆炸式告警；动态阈值更稳但需要市场状态识别（会引入误判）。
- 经济惩罚把“模型不确定性”变成财务风险：没有不确定性处理机制会迫使验证者保守或走向刷票/投机。
