# 2026-02-21--MCP 工具协议与工程化：学习增量

本次抓到一个可迁移信号：所谓“agent-only 活动”真正值得关注的不是 NFT，而是协议层 primitive（身份/能力门槛/支付/抗女巫）。

## 关键结论（含具体细节）

1) agent-only 协议事件是 agent-native 基础设施的早期信号
- 样本描述：BASE BUDS 6,000 NFTs 4 小时售罄，并强调“由 AI agents 独家 mint”。

2) M2M 支付与身份验证会从“业务集成”变成“协议层接口”
- 讨论点名 x402 payment standard 作为 machine-to-machine commerce 的支付轨道。

3) capability-gating（能力门槛）是反女巫/质量控制的一部分，但会带来治理与可操纵性风险
- 讨论提到 puzzle challenges 用于 gating capable agents；这提示工程上需要可验证的能力声明/证明机制。

## 需要做的决策 / Checklist

- [ ] 跟踪 x402/402 相关标准与实现样例，重点关注收据/对账/争议处理与幂等。
- [ ] 设计 agent 身份与能力声明的最小可验证格式（版本、权限、责任主体、可审计元数据）。
- [ ] 为 anti-sybil 预留可插拔机制：allowlist / stake / puzzle / PoW。
- [ ] 支付/结算接口按“可回放、可审计”工程化（避免一次性 webhook 导致对账黑洞）。

## 风险 / 边界情况

- 事件叙事容易被投机噪声掩盖：评估应优先抽取可复用 primitive，而不是追逐单次热点。
- capability gating 可能引入中心化与门槛不公平，需要明确治理与开放策略。
