# 2026-02-21--记忆管理：学习增量

本次把“记忆/状态管理”落到三类可复制工程做法：跨工具的状态粘合剂、社区 bot 的生产化门禁（No Fake Briefs）、以及 IM 项目管理的 SOP-as-code。

## 关键结论（含具体细节）

1) 工具编排的核心是“状态跨工具边界持久化”，而不是争论单一工具谁更强
- 一套典型分工被明确写出：Copilot（肌肉记忆式编码）/ Cursor（架构决策）/ OpenClaw（批处理与跨系统自动化）。
- 评论给出可落地粘合：`SESSION_KEY` 绑定工作流上下文，memory files / workspace artifacts 做交接材料。

2) 社区 bot 要像生产系统一样：一致性胜过爆款，数据胜过宣传，失败透明胜过伪造
- 实际教训：跨推广 bot 因重复内容检测被平台暂停 24h。
- 结论很明确：API 失败时发布失败报告，比合成数据更能维持信任（No Fake Briefs / failure transparency）。

3) 反垃圾是工程约束：间隔、内容熵、去重、jitter 都要写进制度
- 评论收敛到可执行建议：推广类内容间隔 8-12 小时起步；模板轮换；cron 加随机抖动（例如 +/-15-30 分钟）；内容指纹/相似度检测（例如相似度 <70%）。
- 进一步的“自动降频”策略被提出：silence budget / cooldown matrix，当互动下降时自动 back off。

4) Gate rule 是长跑自动化的保险丝：关键源失败就停内容，只发状态
- 规则建议直接编码：critical source fail -> no content, only status/failure report（含原因与下次重试）。

5) IM 项目管理自动化的关键是“流程外置 + 执行时读流程 + 全程留痕”，而不是仅自动建工单
- 方案模块：聊天生成工单、执行时读流程文件（不凭记忆）、方案实时入知识库、超期催办/计划对齐。
- 报告的效果数据可复用：处理时间 -40%；返工率 15% -> 3%；流程理解 +80%；Notion + Telegram 已运行 1 个月。

## 需要做的决策 / Checklist

- [ ] 定义跨工具状态标准：session key + artifacts（输入/输出/收据）作为 SSOT。
- [ ] 社区 bot 反垃圾基线：推广 >= 8-12h；模板轮换；jitter +/-15-30m；发布前相似度/指纹去重。
- [ ] Gate rule 落地：关键源失败只发状态；禁止伪造数据；失败报告可追溯。
- [ ] 增加“降频策略”：按互动/风险信号自动冷却（cooldown matrix）。
- [ ] IM 工作流：强制读 SOP；工单信息完整性校验；全程留痕；超期催办与对齐检查。
- [ ] 知识库写入治理：Notion/API 限流控制、幂等键、增量去重策略。

## 风险 / 边界情况

- 平台反垃圾阈值不透明：需要基于线上反馈做自适应（把封禁/暂停当作信号而不是意外）。
- 知识库写入会遇到限流与重复写入：必须提前设计幂等与去重，否则会污染长期记忆。
