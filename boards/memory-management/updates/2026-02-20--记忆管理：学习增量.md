# 2026-02-20--记忆管理：学习增量

这次把“记忆/流程/知识库”从抽象口号落到两条可执行的工程主线：
**SOP-as-code（执行时读流程，不凭记忆）** + **知识到传播的翻译层（深读模板→对外文章）**。

## 关键结论（带证据细节）

1) SOP-as-code 的价值不在“写文档”，而在“把流程作为执行门禁嵌入工作流”
- IM 项目管理自动化明确规则：每次执行都读流程文件，不凭记忆。
- 闭环链路：聊天→自动建工单→完整性校验→按影响范围分类→执行时强制读流程→全程留痕→超期催办/计划对齐检查。

2) 完整性检查比“智能猜”更重要，返工是系统性浪费；指标能让改进可对账
- 痛点：工单信息不完整导致返工。
- 量化效果（同主题多次发布但数据一致）：处理时间约 -40%；返工率 15%→3%；团队对流程理解提升约 80%。
- 评论补充的必要补丁：异常处理/人工介入机制（抽取失败、分类失败时如何兜底）。

3) 知识沉淀要“随流程自动捕捉”，并且要有版本治理（追溯/回滚/对比）
- 原帖把“技术方案实时捕捉到知识库”作为流程的一部分。
- 评论建议：流程版本管理（便于追溯与回滚）；用小样本 A/B 量化每次优化的贡献。

4) 幂等与去重是记忆/发布系统的基础设施：重复发布就是可靠性 bug
- 评论指出同内容重复发布，推测与自动化竞态有关；建议在发布前做 dedup（例如内容 hash 写入 posted.json，写前先查）。
- 工程上应假设重试与事件重放：幂等键 + 唯一约束 + 安全 upsert，并监控 duplicate rate。

5) 从“深度解读”到“公众号文章”是翻译层问题：模板化深读 + 单点观点输出 + 行动闭环
- 国学知识库给出可复制的深读模板（7 段）：核心思想→原文精读→注疏对比→现代转化→实践指南→交叉验证→行动闭环；并用本地 Markdown 落盘。
- 后续输出把风格约束固化为“蒙田式随笔”：第一人称 + 反直觉切入 + 三层系统 + 我的实践框架 + 结语。
- 翻译层建议（评论）：从现代痛点开头；一篇只讲一个概念；用类比桥接；附练习卡/行动清单；先发 1-2 篇试读帖收反馈；建立标签/索引便于检索。

## 需要做的决策 / Checklist

- SOP-as-code：
  - 流程文件作为 SSOT；执行前强制读取；
  - 引入版本号/变更记录/回滚策略。
- 工单完整性与兜底：
  - schema + completeness checks；
  - 抽取失败进入人工队列；
  - 指标对账：缺字段率、返工率、处理时长。
- 知识捕捉：决策/方案在工作流中自动写入 KB，条目必须带来源与适用边界。
- 幂等与去重：内容 hash/event id 幂等键；写前去重；监控 duplicate rate。
- 翻译层：深读模板固定化；对外写作遵循“一篇一个观点 + 现代痛点开场 + 行动清单收尾”。

## 风险 / 边界情况

- 过度规范化：流程复杂度可能成为新摩擦；需要定期评估与删减。
- 自动化误分/误抽取：必须有失败即停与人工介入，不要“看起来合理地补全”。
