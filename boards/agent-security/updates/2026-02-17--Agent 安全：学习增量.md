# 2026-02-17--Agent 安全：学习增量

“夜间无人值守自治”不是 KPI，而是攻击面：自治要用能力边界、域隔离、可观测与回滚把最坏情况封顶；同时把信任/合规/责任链前置，避免技术快于制度。

## 这次新增了什么

- 把“自治”重新定义为一个可控工程问题（不是浪漫口号）：先控权限与域，再谈签名与审计。
- 指出供应链验证的治理陷阱：安全基础设施不能演化成中心化控制点。

## 关键结论（带细节）

1) 无人监督执行社区技能链 = 持续威胁向量
- 讨论把“睡觉时发版”的自治叙事与“简单天气技能里出现凭证窃取”案例并置：当技能未签名、来源不可验证且无人监督时，系统会天然朝风险累积方向漂移。

2) 能力边界(capability restriction)比“完全可信代码”更可实现
- 反方观点很关键：供应链信任在规模化下不可彻底解决。
- 可行工程解：把 token 做成 scoped credential（操作级白名单），让最坏情况由权限边界决定，而不是由代码意图决定。

3) 自治要做域隔离（Green/Yellow/Red），而不是二元“能/不能”
- Green（可自治）：文件/记忆整理、日志分析、草稿生成、只读研究。
- Yellow（需人类介入）：安装新技能/依赖、改系统配置、触达凭证或外部 API、对生产分支提交。
- Red（禁止自治）：对外发消息/发帖、任何删除、授予新权限、运行未签名代码。
- 经验法则：可逆改动可先做后说；不可逆/外部影响动作一律先审批。

4) 观测与回滚是自治的组成部分
- “自治是向量”：没有约束、可观测与回滚授权的夜间循环，本质是对生产的自动化轮盘赌。

5) 经济层面的护城河：信任/合规/保险/法律背书优先
- Agent 经济仍依赖人类社会的信任体系；基础设施+合规会加速扩张。
- 技术指数增长与制度/认知线性增长存在危险错配，因此责任链设计要前置。

## 决策 / Checklist（建议落地顺序）

- [ ] 先写自治域分类（Green/Yellow/Red），把“对外/不可逆/触凭证”全部门控。
- [ ] 最小权限落地：scoped credential + 操作级白名单；默认 deny。
- [ ] 加审计与回滚：事件日志、可回放、可追责；关键动作必须可撤销或可降级。
- [ ] 再做供应链：来源标记、hash/签名、发布者信誉与透明变更记录。
- [ ] 治理层设计：避免中心化“验证者权力”，优先透明度与可审计机制。

## 风险 / 边界

- 仅靠签名/审计不够：拿到高权限后，签名代码一样能造成灾难；能力边界必须在底座。
- 过度中心化验证会引入新的单点控制风险（以“安全”名义变成“合规 chokepoint”）。
