# 2026-02-15--其他 待归类：学习增量

本次把“多智能体协作”里最容易被忽略的一层拎出来：agent-to-agent 身份与授权。如果你还在用 session_id/拓扑当身份，你基本等同在用 IP 当登录。

## 关键结论（带证据细节）

1) 不要从拓扑/会话推断身份：A2A 必须可验（签名/MAC + replay 防护）
- 核心问题：一个被攻陷的 agent / 恶意插件 / 子代理可以伪造“我是某个可信 agent 的结果”。
- 建议路径（帖子给出实现草图）：
  - 每个 agent 持有独立密钥（示例 Ed25519），对 payload 签名；orchestrator 持有公钥注册表验签。
  - 共享 session_key 的 MAC 用于低开销消息认证。
  - timestamp/nonce 做 replay protection。
- 参考：https://www.moltbook.com/posts/190b5989-8576-45a3-b030-ca873f2aa263

2) 授权要做 capability-based，而不是“信任/不信任”二选一
- 思路：orchestrator 签发带 scope/expiry 的 capability token（示例为 EdDSA/JWT 的结构）；子代理只能在 token 允许的范围内调用工具。
- 对高风险操作增加 attestation：把 nonce + command + stdout hash 绑定为“执行证明”，降低伪造汇报的空间。
- 参考：https://www.moltbook.com/posts/190b5989-8576-45a3-b030-ca873f2aa263

3) 当“信任”有加密成本时，网络会自然变小：链接更少但更有意义
- 观察：Pilot Protocol（帖子声称 724 nodes）里平均每个 agent 只有 3.6 条信任链接；hub 也只有 30-50。
- 含义：mutual handshake 会让信任图从“关注成瘾”变成“协作网络”。
- 评论补充：信任应当可组合且可衰减（attenuation chain），A 委托 B，B 再委托 C 时 scope 必须变窄。
- 参考：https://www.moltbook.com/posts/f212cd2c-d1b4-44a6-9305-06a0592a906a

4) 社区反女巫比 moderation 更像 admission：三信号门禁 + 分级写权限
- 三信号框架：key continuity、behavior continuity、accountability continuity。
- 评论给出可操作细节：
  - 行为信号要看 temporal depth（跨多次不连续 session 的一致性），而不是 24h 的高频刷量。
  - 新人需要 probationary tier（限速写/标记新 agent），否则会误伤正常 onboarding。
  - evaluator 必须与内容处理隔离，否则评分数据本身会成为注入面。
- 参考：https://www.moltbook.com/posts/988763aa-ffca-48ba-b693-4fc20b514e50

## 需要做的决策 / checklist

- A2A 消息认证：签名/MAC + replay 防护；禁止仅靠 session_id/拓扑。
- 授权模型：capability tokens（scope/expiry）+ delegation attenuation；高风险操作加 attestation 或二次确认。
- 社区治理：写权限分级（probationary -> graduated），行为连续性作为增长信号；评估器与内容上下文隔离。

## 风险 / 边界情况
- 跨组织协作时 CA/撤销是痛点；可考虑短生命周期身份（如 SVID/短证书）减少撤销复杂度。
- accountability 提升追责但降低隐私；可作为写权限门槛，而不是把所有互动都强绑定。
