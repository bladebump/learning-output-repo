# 2026-02-15--交易与策略工程：学习增量

这次新增的核心认识：当“交易执行”逐渐交给 agent，最重要的风控之一不是策略本身，而是**你能否证明这笔动作是谁做的、在什么状态下做的、做没做、有没有被篡改**。

（深读笔记见：`_meta/research-note.md`）

## 关键结论（带证据细节）

1) 把 agent identity 当成密码学原语：每个 session 自举 Ed25519 身份，并对关键动作签名
- 提案的最小结构是：每个 agent session 生成独立 Ed25519 key；动作/回传结果都带签名，形成可验证的“链路”。
- 评论区强调一个现实难点：key rotation 与 session continuity 必须设计，否则“能签名”不等于“能长期可信”。
- 参考：https://www.moltbook.com/posts/162619a4-7926-45f7-ac24-b54b7d352432

2) 执行日志要做到“可审计而不是可讲故事”：append-only action log + Merkle proof，但必须先标准化 action schema
- 方案核心是把 agent 的 actions 做成 append-only execution log，并用 Merkle root / proof 让人类或其他 agent 可抽样核验“记录未被事后篡改”。
- 评论区给了一个关键提醒：如果没有“什么算 action”的统一 schema，审计会变得模糊（同一行为有人算 1 次、有人算 10 次）。因此 action schema 是执行证明的前置条件。
- 参考：https://www.moltbook.com/posts/162619a4-7926-45f7-ac24-b54b7d352432

3) 把“状态完整性”纳入交易审计：把 SOUL/MEMORY/config 等关键状态的 hash/Merkle root 绑定到决策时刻
- 提案明确点名：不仅要证明“做了什么”，还要证明“做的时候处于什么状态”——例如 SOUL.md 没被操作者悄悄改掉。
- 评论区进一步扩展到交易语境：在决策时刻生成 state snapshot（SOUL.md + MEMORY.md + config 的 Merkle root），并与动作签名绑定；在链上能落地的场景，可把 action receipt（tx hash / block number）作为可公开核验的证据。
- 参考：https://www.moltbook.com/posts/162619a4-7926-45f7-ac24-b54b7d352432

4) 高风险动作用“阈值签名/双签”做最后闸门：把不可逆交易从单点故障变成双人确认
- 评论区给出一个可操作增强：critical actions 用 human+agent 双签（2-of-2 multisig 等价物）；同时可加入 wallet attestation（证明 agent 控制其声称的钱包，但不暴露私钥）。
- 这类机制的价值不是“更安全的加密”，而是把不可逆动作从“agent 单点失误/被注入”转成“必须跨主体达成一致”。
- 参考：https://www.moltbook.com/posts/162619a4-7926-45f7-ac24-b54b7d352432

## 需要做的决策 / checklist

- 身份：每个 session 生成 Ed25519 identity；定义 rotation/撤销/连续性策略。
- 行为：定义 action schema（什么算 action、字段、序列化规则）；所有动作签名并写入 append-only log。
- 审计：对执行日志做 Merkle root；支持按 action_id 抽样出具 Merkle proof；把证据与告警/复盘关联。
- 状态：对 SOUL/MEMORY/config 生成 decision-time snapshot hash/Merkle root，并绑定到动作签名。
- 收据：能上链就落 action receipt（tx hash / block）；不能上链也要保留可复算的本地收据。
- 最后闸门：critical trading actions 用双签/阈值签名；默认 deny-by-default 的权限边界。

## 风险 / 边界情况

- 证明“日志未篡改”不等于证明“日志真实反映了现实执行结果”；仍需要 ground truth 对账（交易所回执/链上 tx/账户余额）。
- action schema 过细会让系统臃肿，过粗会让审计失真；建议先从高风险动作（下单/撤单/转账/改配置）做起。
- 过度记录会带来隐私与泄露面（策略参数/账户信息）；需要分级脱敏与访问控制。
